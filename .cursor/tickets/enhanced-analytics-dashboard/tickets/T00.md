## Description

| Deliverable | Detail |
|-------------|--------|
| **Grain** | Closed calendar days only in **Europe/Berlin** (CET/CEST): `bucket_date < current_date` in a Berlin session (or equivalent). **Exclude “today”** from the MV — document in SQL comment and/or short README. |
| **Dimensions** | Daily row per **(organization_id, machine_id)** (both **text**, not UUID). Align column names/types with existing `purchase` / org / machine schema. |
| **Measures** | Purchase aggregates (e.g. count, `sum(amount_in_cents)`), **revenue share** from **contract version at time of purchase** (`revenueShareBasisPoints` → `floor(amount * basisPoints / 10000)` per purchase unless product standard says otherwise). Optional: JSON breakdowns for product / business entity — **or** defer breakdowns to separate MVs / live queries; **document the choice** in migration comment or `packages/db/docs/` / `tickets/README-analytics-mv.md`. |
| **Rent / operator revenue** | If the wide MV includes prorated rent / operator revenue, follow plan §1.1 (per-machine monthly rent prorated by Berlin calendar days; split on contract version changes). |
| **Indexes** | **UNIQUE** index on the MV grain (required for `REFRESH MATERIALIZED VIEW CONCURRENTLY`). Supporting B-tree indexes for `(organization_id, bucket_date)` and `(machine_id, bucket_date)` (or equivalent query patterns). |
| **Migration** | Raw SQL migration (Drizzle-generated or hand-written per project convention) creating the MV + indexes. |
| **Documentation** | Do not leave temporal / DST logic undocumented — cross-link MV README or `packages/db` doc snippet. |

### Product / technical rules (echo)

- All bucket boundaries and “today” use **Europe/Berlin**; periods are **calendar** day / week / month, not rolling windows.
- **IDs are `text`** everywhere in MV definitions and indexes.
