## Description

Transform the admin app's protected layout to: (1) change the auth gate so non-admin signed-in users see a "role needs update" block screen instead of redirecting to sign-in; (2) create an admin-specific app shell with a sidebar composing base components from `packages/ui` (no TeamSwitcher); (3) add NavMain links for Users, Customers, Contracts, Machines; (4) add placeholder routes for customers, contracts, machines; (5) use NavUser in the sidebar footer with `onSignOut` only (admin always shows "Sign out"); (6) integrate the sidebar layout so it replaces or complements the root header for `_admin` routes.

**Step-by-step implementation:**

1. **Auth gate change:** In `apps/admin-frontend/src/routes/_admin/route.tsx`, update `beforeLoad`:
   - If `!session.data`, redirect to `/sign-in` (existing behavior).
   - If `session.data.user.role !== "admin"`, do NOT redirect. Instead, return context or throw to a route that renders the role block screen. Option: create a route `_admin/role-block` or render the block inline when role check fails. Per plan: "non-admin → role block screen" — the layout can conditionally render either the block or the outlet based on role.
   - Suggested approach: In `beforeLoad`, if not admin, return `{ blocked: true }`. In the layout component, if `blocked`, render the role block UI; else render the sidebar + Outlet.
   - Remove the duplicate/redundant role check (current code has two similar checks).

2. **Role block screen:** Create `apps/admin-frontend/src/routes/_admin/role-block.tsx` or a shared component `RoleBlockScreen`. Message: "Your account does not have admin access. Contact your administrator to request admin privileges." Static, no self-service flow.

3. **AdminAppSidebar:** Create `apps/admin-frontend/src/components/admin-app-sidebar.tsx`. Compose from `packages/ui` base:
   - `Sidebar` (collapsible="icon"), `SidebarContent`, `SidebarFooter`, `SidebarRail`
   - No `SidebarHeader` with TeamSwitcher
   - `NavMain` with items: Users (`/users`), Customers (`/customers`), Contracts (`/contracts`), Machines (`/machines`)
   - `NavUser` in footer with `user={session.data.user}` (map to { name, email, avatar }), `onSignOut={() => authClient.signOut()}`
   - Reference `packages/ui/src/base/app-sidebar.tsx` structure but do NOT import it — compose from `Sidebar`, `SidebarContent`, `SidebarFooter`, `NavMain`, `NavUser` directly.

4. **Layout structure:** The `_admin` layout component should wrap with:
   ```tsx
   <SidebarProvider>
     <AdminAppSidebar />
     <SidebarInset>
       <SidebarTrigger />
       {blocked ? <RoleBlockScreen /> : <Outlet />}
     </SidebarInset>
   </SidebarProvider>
   ```
   The root has a header; for `_admin` routes the sidebar is the main chrome. Either hide the root header when inside _admin (via context/layout) or make the _admin layout full-bleed. Per plan: "sidebar replaces or complements header" — simplest is to have _admin layout fill the main area with sidebar + content, and the root header may stay for sign-in; for _admin the content area shows the sidebar shell.

5. **Placeholder routes:** Create:
   - `apps/admin-frontend/src/routes/_admin/customers.tsx` — placeholder page (e.g. "Customers — Coming soon")
   - `apps/admin-frontend/src/routes/_admin/contracts.tsx` — placeholder
   - `apps/admin-frontend/src/routes/_admin/machines.tsx` — placeholder
   Add NavMain items linking to `/customers`, `/contracts`, `/machines`. Users already exists at `/users`.

6. **Route config:** Ensure `_admin/index` redirects to `/dashboard`. Nav links: Users → `/users`, Customers → `/customers`, Contracts → `/contracts`, Machines → `/machines`.

## Research / Context7

### TanStack Router layout routes, beforeLoad

```tsx
export const Route = createFileRoute("/_admin")({
  beforeLoad: async () => {
    const session = await authClient.getSession();
    if (!session.data) {
      throw redirect({ to: "/sign-in", throw: true });
    }
    if (session.data.user.role !== "admin") {
      return { blocked: true }; // pass to component
    }
    return { blocked: false };
  },
  component: AdminLayoutComponent,
});
```

Use `throw redirect()` for unauthenticated; return context for role-block case. Component reads context and conditionally renders.

### Shadcn Sidebar Composition

```tsx
import { SidebarProvider, SidebarInset, SidebarTrigger } from "@slushomat/ui/base/sidebar";
import { NavMain } from "@slushomat/ui/base/nav-main";
import { NavUser } from "@slushomat/ui/base/nav-user";

<SidebarProvider>
  <AdminAppSidebar />  {/* Sidebar + SidebarContent + SidebarFooter + NavMain + NavUser */}
  <SidebarInset>
    <SidebarTrigger />
    {children}
  </SidebarInset>
</SidebarProvider>
```

NavMain expects `items` with `{ title, url, icon?, isActive?, items? }`. Use flat structure for top-level links:
```tsx
const navItems = [
  { title: "Users", url: "/users", icon: <UsersIcon /> },
  { title: "Customers", url: "/customers", icon: <UsersIcon /> },
  // ...
];
```

## Risk / Notes

- **Root header:** The root has `<header>Slushomat Admin</header>`. For _admin routes, the sidebar provides navigation. Decide: (a) hide root header when path is under _admin, or (b) keep both (header + sidebar). Plan says "sidebar replaces or complements header" — frontend-agent can hide the header in _admin or make the layout full-bleed. Simplest: root keeps header; _admin layout fills the main with sidebar + content. The header might still show "Slushomat Admin" — acceptable unless requirements say otherwise.
- **NavMain structure:** The base `NavMain` uses Collapsible with nested items. For flat nav (Users, Customers, etc.), either pass items with no nested `items` or use `SidebarMenu` + `SidebarMenuButton` + `Link` directly. Check nav-main.tsx structure.
- **Session in layout:** Admin layout needs `session.data.user` for NavUser. Fetch in beforeLoad and pass via route context, or use a hook/query in the layout component.
