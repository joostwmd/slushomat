## Acceptance Criteria (traceable)

This task does not implement product ACs directly; it **clarifies** how AC-1–AC-5, AC-9, and AC-10 should behave in the UI.

| AC ID | Requirement area | What the UI spec should nail down |
|-------|------------------|-----------------------------------|
| AC-1 | Products list | Layout, columns, thumbnail vs placeholder, empty state |
| AC-2 | Create product | Form fields, validation messaging, success feedback |
| AC-3 | Image upload | End-to-end steps (after save?), progress, success |
| AC-4 | Edit product | Same sheet vs separate pattern; dirty state |
| AC-5 | Delete | Confirm dialog; expectation for storage + DB (align with human checkpoint 1) |
| AC-9 | Upload failure / orphan | Retry upload vs delete orphan product — explicit UX |
| AC-10 | Invalid file | Pre-upload vs post-error; max size / types if not only server-side |

## Context from Discovery

### Requirements (`00-requirements.md`)

- Admin route **Products** (e.g. `/products` under `_admin`).
- **Dice UI** file upload (`@diceui/file-upload`) aligned with `@slushomat/ui` and Tailwind.
- **MVP:** one image per product; schema may allow more later — spec should state **single-image MVP** vs gallery for launch.
- **AC-9:** “Upload failure after product create: user can retry upload or remove orphan product (define UX in UI spec).”
- **AC-10:** “Invalid file type / size: rejected before or during upload with clear feedback.”

### Plan (`03-plan.md`)

- List + **sheet/dialog** for create/edit.
- **Image replace** flow.
- **Error states:** upload fail, invalid file.
- **Empty state**, **delete confirm**.
- **Dashboard wording:** keep `/dashboard` as-is vs redirect `/dashboard` → `/products` — decide at human checkpoint 1 and record in spec.
