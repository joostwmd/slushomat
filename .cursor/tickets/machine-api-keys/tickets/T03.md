## Product rules (carry through implementation)

1. **Keys:** Better Auth plugin, prefix **`SLUSH_`**, dedicated **`configId: "machine"`**, **`metadata.machineId`**, **`userId` = authenticated admin** on `createApiKey` (supported by this plugin config).
2. **One key per machine** — rotation deletes/replaces the previous key; revoke clears **`machine.apiKeyId`** (enforced in T04/T05).
3. **Never** return full plaintext secret except create/rotate responses; list/get metadata only (T05/T06).
4. **Machine server:** Bearer + **`X-Machine-Id`** → `verifyApiKey` → match metadata to machine id → load **`machine`** → **`disabled` → 403** `MACHINE_DISABLED`, else invalid → **401** `INVALID_MACHINE_CREDENTIALS` (implemented in T05).
5. Shared **`auth`** from **`@slushomat/auth`** in machine-server (T05: dependency + env).

## Description

Wire **Better Auth’s API Key plugin** (`@better-auth/api-key`) into the shared **`auth`** singleton so machine credentials are stored **hashed** in a dedicated **`apikey`** table, with a **`configId: "machine"`** profile: prefix **`SLUSH_`**, **metadata** enabled (for `metadata.machineId`), **`references: "user"`** (issuer = admin `userId` at create time). Configure header extraction so **`verifyApiKey` / plugin** accept secrets sent as **`Authorization: Bearer <token>`** (not only `x-api-key`). Run the repo’s **`pnpm auth:generate`** flow, merge generated Drizzle definitions into **`@slushomat/db`**, and commit SQL migrations so CI and local DBs match production.

## Implementation steps

1. Add **`@better-auth/api-key`** to **`@slushomat/auth`** dependencies; run install from monorepo root.
2. In **`packages/auth/src/index.ts`**, add **`apiKey({ ... })`** to **`plugins`** after reviewing [API Key plugin](https://www.better-auth.com/docs/plugins/api-key) and [Reference](https://www.better-auth.com/docs/plugins/api-key/reference) for the exact option names your **better-auth** version expects (array vs single **`config`** / **`configId`**).
3. Set **`enableMetadata: true`**, **`defaultPrefix: "SLUSH_"`**, **`references: "user"`**, and a dedicated **`configId: "machine"`** (or equivalent nested config).
4. Configure **`startingCharactersConfig`** (or equivalent) so list/UI can show a short **`start`** prefix of the key for identification.
5. Configure **`apiKeyHeaders`** and/or **`customAPIKeyGetter`** so the **raw secret string** passed to verification is taken from **`Authorization: Bearer …`** (strip `Bearer ` case-insensitively).
6. Run **`pnpm auth:generate`** (root script targets `packages/auth/src/index.ts` via `apps/server/.env` for `DOTENV_CONFIG_PATH`).
7. Merge generated Drizzle schema into **`packages/db`** so **`drizzleAdapter(db, { schema })`** includes **`apikey`**; fix any naming (`apikey` vs `apiKey` export) to match what the adapter expects.
8. Generate/commit migration SQL; verify **`pnpm db:migrate`** (or team equivalent) applies cleanly on a fresh DB.

## Code examples (illustrative — verify against installed package versions)

### 1) `packages/auth/src/index.ts` — plugin registration (pattern)

```typescript
import { apiKey } from "@better-auth/api-key";
// ... existing imports: betterAuth, drizzleAdapter, admin, organization, oneTimeToken

export const auth = betterAuth({
  database: drizzleAdapter(db, { provider: "pg", schema }),
  // ...existing emailAndPassword, session, databaseHooks, advanced
  plugins: [
    admin(),
    oneTimeToken(),
    organization({ /* unchanged */ }),
    apiKey({
      enableMetadata: true,
      defaultPrefix: "SLUSH_",
      references: "user",
      // If your version uses a keyed config map / array, set configId: "machine" there.
      // startingCharactersConfig: { shouldStore: true, length: 4 }, // tune per docs
      customAPIKeyGetter: (ctx) => {
        const raw = ctx.request?.headers.get("authorization");
        if (!raw) return null;
        const m = /^Bearer\s+(.+)$/i.exec(raw.trim());
        return m?.[1]?.trim() ?? null;
      },
    }),
  ],
});
```

*If your Better Auth version prefers **`apiKeyHeaders`** instead of **`customAPIKeyGetter`**, use the documented combination that still yields Bearer secrets — the invariant is: **machine-server sends `Authorization: Bearer <key>`** and verification must accept it.*

## Pitfalls / notes

- **Generator output location**: CLI may emit a new file (e.g. `apikey.ts`); ensure **`packages/db/src/schema/index.ts`** exports it and **`drizzleAdapter`** receives the full schema object.
- **Table / column names**: Better Auth’s Drizzle schema may use **`apikey`** as the table identifier; Drizzle-ORM relations to **`machine.apiKeyId`** happen in **T04** — avoid circular imports (sometimes FK is deferred as plain `text`).
- **Do not** enable **`disableKeyHashing`** except possibly local scratch — violates **AC-7**.
- **Context7** was unavailable at ticket time; treat snippets as **patterns** and lock to the installed **`better-auth` / `@better-auth/api-key`** types.
