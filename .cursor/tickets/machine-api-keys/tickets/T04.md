## Product rules (carry through implementation)

1. **Keys:** Better Auth plugin, prefix **`SLUSH_`**, **`configId: "machine"`**, **`metadata.machineId`**, **`userId` = admin** at create (T03/T05).
2. **One key per machine** — **`machine.apiKeyId`** holds the single active key id; rotation/revoke semantics in T05.
3. **Never** expose plaintext secret from DB columns — only hashed storage in **`apikey`** (T03).
4. **Machine server** auth behavior is specified in T05; this ticket only supplies **`disabled`** + **`apiKeyId`** on **`machine`**.
5. Shared **`auth`** from **`@slushomat/auth`** (T03/T05); DB layer stays in **`@slushomat/db`**.

## Description

Extend the **`machine`** Drizzle model with an operational **kill switch** and a **nullable link** to the active Better Auth API key row: **`disabled`** (`boolean`, default **`false`**) and **`apiKeyId`** (`text`, nullable). Add an **index** on **`apiKeyId`** if admin or jobs query by it. Prefer a **foreign key** to the generated **`apikey.id`** when the **`apikey`** table lives in the same schema package without import cycles; otherwise use **nullable `text` + application-level integrity** and a short comment explaining why. Run **`drizzle-kit generate`**, commit migration SQL, and keep **`machineRow`** / API outputs in sync for downstream tasks.

## Pitfalls / notes

- **Migration order**: **`apikey`** table migration (T03) should run **before** any FK from **`machine.api_key_id`** to **`apikey.id`** — combine or order migrations accordingly.
- **Postgres column naming**: Drizzle `apiKeyId` → often snake_case **`api_key_id`** in DB; stay consistent with existing **`machine_version_id`** style.
- **Default for existing rows**: **`disabled = false`** for all current machines.
- **Do not** store plaintext secrets on **`machine`** — only the **key id** reference.
